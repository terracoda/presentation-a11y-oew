Assistive & Adaptive Technologies

A range of devices and software that help people with disabilities do things that they would not otherwise be able to do.

Diverse range of Assistive Technologies

Often highly customized for a person's disability.

This is where the digital realm has an advantage over the physical realm. Our medium, the digital realm, allows us to customize for small groups of people, even customize to one person.

So we have some awareness

Let's start making content accessible one step at time. We need to employ special techniques to build-in the blue part - to turn the stairs into a ramp.

Building the virtual ramp

Our goal in WCAG 2.0 language is to make our web content:

  1. Perceivable

  2. Operable

  3. Understandable

  4. Robust

(WCAG 2.0, 2008)

The access techniques that I will now describe focus on making web content accessible one media type at a time – one step at a time.

I'm intentionally not talking about exact WCAG success criteria – our goal is accessibility not just compliance. Each step may correspond to more than one success criteria.

A person is not going to sue you because you failed on a particular success criterion, but because he/she could not access something really important on your website (likely multiple times).

Most people who have access problems don't sue. Most people just have a bad experience and never come back.

Team effort

  • It's about the content – the range of media types from basic text to interactive exercises and online video.

0. A solid base: good code

Three essential things are needed for a solid (valid) code base.

  1. Document Type

    <!DOCTYPE html>

    The doctype tells the validator & the browser what version of HTML the site is coded in.

  2. A Page Title

    <title>Unique Page Title Goes Here</title>

    A unique and meaningful page title tells users where they are. The title element is required.

  3. Character Encoding

    <meta charset="UTF-8">

    If you have ever seen a web page with funny little symbols instead of letters and punctuation, it was likely having an encoding problem.

1. Text: accessible by nature

  1. Declare the language

    The browser needs to know what language the content is written in, and it needs to made aware of language changes.

    <html lang="en">

    <q lang="fr">Bonjour!</q>

  2. Logical & Meaningful Structure

    Headings, paragraphs, lists, quotations, emphasis, etc.

  3. Natural linear order

    With style and scripts turned off, a user should be able to read the content from top to bottom (depending the language, of course).

  4. Reading Level

    Keep it as simple as possible.

{demo}

2. Links & Navigation

The stepping stones of the internet. Well-planned, consistent navigation benefits all users.

  1. Skip to Content links

    Special links (often hidden) that provide a direct route to the main sections or landmarks.

    HTML5 has a robust set of new landmark tags.

  2. Consistent navigation

    Users love consistency as it usually makes things easier to use.

  3. Unique links & Meaningful words

    • DRY principle: Do not repeat links to the same place over and over.
    • Clear function: Indicate the function of the link and avoid generic Read more statements.
    • User control: Leave context changes in the control of the user (ie no forced new tabs).

{demo}

3. Images & Charts

A text-based alternative to all meaningful visual content is required.

  1. Short descriptions

    <img src="image-file.jpg" alt="Description." >

    Short descriptions can go in:

    • alt attribute (hidden to visual user)
    • caption (figcaption in html5)
    • content (visible to everyone)

  2. Complex images

    May need longer descriptions than is appropriate for the alt attribute.

    • Link to a supplementary description.
    • Maintain the meaning of the document.

{demo}

4. Tables

Use the rich set of semantic table mark-up that is available.

  1. KISS principle

    Simple tables benefit all users.

  2. Caption your tables

    <caption>Introduce the table here.</caption>

    Table captions are associated directly with the table and give a visually impaired user a brief description of the data and the layout.

  3. Scope out direction

    <th scope="row">Describe the table here.</th>

    Use the scope attribute to give the row or column content direction.

  4. Tabular data only

    Use CSS for layout.

{demo}

5. Design Choices

Accessible design goals need to start at the beginning of project.

  1. Use CSS for Layout

    Using CSS & HTML together properly gives you the biggest bang in making your content accessible.

  2. Choose Colours Wisely

    • Don't rely on colour alone for meaning.

      Use other formatting or icons to communicate important things such as links, state changes and messages.

    • Test for sufficient contrast. Have a minimum standard in place to guide designers.
    • Build awareness and avoid colour confusion. Try the Brewer palette. The WCAG Samurai have advice on using Brewer colour combinations.

6. Forms are complex by nature

Keep things simple. Don't ask for data you don't need.

  1. Use & associate a label for every form control.
  2. Group form controls to create clarity: legend, fieldset, optgroup. Use the rich set of tags for creating meaning in forms is good for all users.

Note: Forms & complex data tables have often been the basis for lawsuits.

  • Maguire v. Sidney Organizing Committee for the Olympic Games (SOCOG) - results table.
  • Canada v. Jodhan - job application form.
